# ADR-0004: `message.id` — ULID в виде `CharField(26)`

**Status:** Accepted
**Date:** 2026-02-28

## Контекст

`Message` — главная горячая таблица. Под ожидаемой нагрузкой 200 RPS пиково (~17 млн строк в сутки максимум) выбор первичного ключа важен: он определяет cluster locality на индексах, размер индексов в B-tree, удобство логирования и trace correlation.

## Решение

`id = CharField(primary_key=True, max_length=26, default=lambda: str(ULID()))`.

ULID — 128-битный идентификатор: 48 бит миллисекундный timestamp + 80 бит криптографического random. В base32-Crockford занимает 26 ASCII-символов. Сортируется лексикографически = по времени.

## Последствия

**+** Lexicographic-sortable = новые записи попадают в конец индекса = последовательные вставки = меньше B-tree splits, меньше WAL по сравнению с UUIDv4.
**+** Включает timestamp — по `id` можно прикинуть время создания без `created_at`, удобно в логах и `X-Notify-Message-Id` хедере.
**+** 26-символьный ASCII легко вставить в URL, лог, копировать. UUID-формат с дефисами визуально хуже.
**+** Нет «угадываемости» соседних id (как у autoincrement) — нельзя scrape'ить чужие сообщения перебором.
**−** 26 байт вместо 8 (bigint) — в индексе и FK. На 17M строк в день это ~430MB в год только под `id` в PK. Приемлемо, но не бесплатно.
**−** Не UUID — не дружит «из коробки» с инструментами, которые ждут UUID-формат (Django Admin, OpenAPI generators). Решено через `CharField` + ручной serializer field.

## Альтернативы

1. **`bigint` autoincrement.** Самый компактный, лучший для B-tree. Минус — id предсказуемы, и кросс-сервисная корреляция требует префиксов. Для внешнего API считаю identifiers-as-API-tokens плохой идеей.
2. **UUIDv4.** Стандарт, но random → разбросанные вставки по всему индексу → выше IO. На наших объёмах ещё ОК, но трендовая практика 2024+ — отказ от v4 для PK горячих таблиц.
3. **UUIDv7.** Тот же time-sortable профиль что и ULID, RFC 9562, нативно `UUIDField`. **Это сильный кандидат**, и если бы стартовали в 2025+ с библиотечной поддержкой, выбрал бы его. Сейчас выбрал ULID, потому что `python-ulid` стабилен 4+ лет, а UUIDv7 в Python ещё фрагментирован (`uuid6`, `uuid-utils`, `uuid7` — разные пакеты, ни один не «официальный»).
4. **Snowflake (Twitter-style).** Требует id-генератора с эпохой и worker-id. Лишняя координация ради тех же свойств.

## Миграционный путь к UUIDv7

Если стандартная библиотека Python получит `uuid.uuid7()` (proposal в PEP-таки), миграция:
1. Меняем default-функцию: `default=lambda: uuid7().hex` (32 hex-символа без дефисов = тот же `CharField(32)`).
2. Старые ULID'ы остаются как есть (валидные ASCII identifiers).
3. Никаких миграций данных.
